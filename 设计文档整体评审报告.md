### 1.3 功能模块分析

实际模块数为 **12 个**（非文档暗示的 11 个），`§3.10a` 生产工单是后插入的编号。

**业务流程闭环分析**：

```
采购全链路：供应商报价 → 采购单 → 审核 → 入库(分批) → 库存↑ → 应付生成 → 付款 ✅
销售全链路：客户下单 → 销售单 → 审核 → 出库(分批) → 库存↓ → 应收生成 → 收款 ✅
退货闭环：  入库/出库单 → 退货单 → 库存回调 → 应付/应收冲减（独立调整记录） ✅
生产闭环：  BOM → 工单 → 领料出库 → 完工入库 ✅
定制闭环：  定制单 → BOM 展算 → 报价 → 转销售单 → 生产工单 → 交付 ✅
补货闭环：  库存预警 → 补货建议 → 一键生成采购单 ✅
```

**闭环缺口**：

| 缺口 | 影响 |
|------|------|
| 🔴 **退货后移动加权平均成本回调规则缺失** | 退货单价与当前平均成本不同时直接扣减会导致成本失真 |
| 🔴 **销售出库类型定义冲突**：枚举包含"生产领料出库"，应属于生产工单模块 | 逻辑混淆 |
| 🔴 **费用分摊尾差处理未定义** | 多次分批入库舍入误差累计后总额 ≠ 订单级费用 |
| 🟡 入库超采购数量的处理规则缺失 | 实际操作中常见，需定义比例上限 |
| 🟡 盘点期间库存变动处理未明确 | 盘点过程中出入库会导致偏差 |
| 🟡 预付款/定金场景缺失 | 家具定制和大宗采购中预付定金高频 |

### 1.4 技术可行性

| 方面 | 评价 |
|------|------|
| Tauri 2 + Next.js 16 | ✅ 适配桌面端；⚠️ Next.js SSR/RSC 优势在静态导出下无法发挥，建议评估 Vite 替代方案 |
| SQLite 单机场景 | ✅ 完全胜任，百万级数据无压力 |
| 🔴 越南语排序/搜索 | SQLite 默认 collation 不支持越南语正确排序，需 ICU 扩展 |
| 🟡 PDF 字体嵌入 | 中文 + 越南语字体可能使安装包超过 30MB 目标 |

## 二、数据库设计评审 (`02-database-design.md`)

### 2.1 数据模型质量

**总计**：46 张表（45 业务 + 1 迁移跟踪）、117 条索引、6 个计算列

| 维度 | 评分 | 说明 |
|------|------|------|
| 范式合规 | ⭐⭐⭐⭐⭐ | 所有表达 3NF+，反范式设计均有明确业务理由 |
| 主键策略 | ⭐⭐⭐⭐⭐ | 统一 `INTEGER PRIMARY KEY AUTOINCREMENT`，v2.0 预留 `BIGSERIAL` |
| 字段命名 | ⭐⭐⭐⭐ | 全局 `snake_case`，少量不一致（`work_order_code` vs `order_no`） |
| 数据类型 | ⭐⭐⭐⭐⭐ | 金额 INTEGER 最小货币单位、数量 REAL、枚举 TEXT+CHECK |
| 约束设计 | ⭐⭐⭐⭐ | NOT NULL/UNIQUE/CHECK/DEFAULT 覆盖全面 |

### 2.2 核心设计亮点

1. **🏆 金额存储方案**：INTEGER 存最小货币单位，彻底避免浮点精度问题，三币种精度系数清晰
2. **🏆 批次追溯模型**：三级追踪模式(none/optional/required) + 批次预留分配 + 全链路 `lot_id`
3. **🏆 毛利双口径**：出库时同时固化标准成本和实际成本快照
4. **🏆 快照策略一致**：操作人名称、单位名称、换算率、成本等历史数据均有快照
5. **🏆 退货冲减设计**：独立 `adjustment_type = 'return_offset'` 负金额记录，不修改原始应付/应收
6. **🏆 v2.0 迁移预留**：ID 策略、SQL 方言隔离、PRAGMA 集中管理、迁移脚本双版本

### 2.3 问题清单

**🔴 高风险**：

| # | 问题 | 建议 |
|---|------|------|
| H1 | 退货/工单自动生成的出入库单缺少源单据回溯字段（`inbound_orders` 缺 `sales_return_id`/`work_order_id`） | 增加 `source_type` + `source_id` 通用字段 |
| H2 | 汇率 `REAL` 存储极小值（VND: 0.000039）精度风险 | 改为"1 USD = N 外币"方向存储 |
| H3 | 全表无外键约束，完全依赖应用层代码质量 | 建议开发/测试环境临时启用 `PRAGMA foreign_keys = ON` |

**🟡 中风险**：

| # | 问题 |
|---|------|
| M1 | `updated_at` 仅有 DEFAULT 无自动更新机制（SQLite 不支持 ON UPDATE） |
| M2 | `system_config` 缺少 `backup_time`、`setup_completed` 等配置项 |
| M3 | `operation_logs` 和 `inventory_transactions` 无归档方案（5 年后达 50 万行） |
| M4 | 部分明细表无 `created_at`/`updated_at` |
| M5 | `stock_checks` 缺少 `warehouse_id` 索引 |
| M6 | 期初库存导入缺少数据标记（无 `initial_import` 交易类型） |

### 2.4 索引改进建议

```sql
-- 建议补充的索引
CREATE INDEX idx_sc_warehouse ON stock_checks(warehouse_id);
CREATE INDEX idx_pay_due ON payables(due_date);
CREATE INDEX idx_recv_due ON receivables(due_date);
CREATE INDEX idx_oo_customer ON outbound_orders(customer_id);
CREATE INDEX idx_io_supplier ON inbound_orders(supplier_id);
CREATE INDEX idx_invt_source ON inventory_transactions(source_type, source_id);
```

## 四、开发计划评审 (`04-development-plan.md`)

### 4.1 阶段划分

5 个迭代阶段遵循合理的依赖递进关系：

```
脚手架/基础 → 主数据 → 核心业务(采购+销售+库存) → 财务报表 → 翻译打磨发布
```

**亮点**：
- 阶段三拆为 3A（采购+库存基础）和 3B（销售+定制+库存高级），降低单阶段复杂度
- i18n 和多币种从阶段一就搭建，避免后期返工 —— 非常正确的策略
- 每阶段有明确的联调缓冲时间和 checklist 格式验收标准

### 4.2 🔴 核心问题：工期估算偏乐观

| 维度 | 文档标称 | 实际评估 |
|------|---------|---------|
| 总工期 | 26-28 周 | **32-36 周**（甘特图串行总工时 ~176d ≈ 35 周） |
| 阶段三B | 4 周 | **6-8 周**（串行约 37d，含定制单+生产工单+销售+库存高级） |
| POC 验证 | 1 天（4 项） | 建议至少 **2-3 天** |
| 假期/病假缓冲 | 未计入 | 单人开发半年+，至少需 1-2 周缓冲 |

### 4.3 各阶段工期重新评估

| 阶段 | 标称 | 评估 | 风险 |
|------|------|------|------|
| 一：脚手架 | 4 周 | 4-5 周 ✅ | 45 张表 DDL + POC 可能超时 |
| 二：基础数据 | 3.5 周 | 5-6 周 ⚠️ | BOM 成本核算复杂；通用组件 1d 偏紧 |
| 三A：采购+库存 | 4 周 | 4.5-5.5 周 ⚠️ | 多币种+批次+费用联动 |
| **三B：销售+定制** | **4 周** | **6-8 周** 🔴 | **全项目最复杂阶段** |
| 四：财务+报表 | 3.5 周 | 4.5-5.5 周 ⚠️ | 费用分摊、双口径毛利 |
| 五：翻译+发布 | 4.5 周 | 5-6 周 ⚠️ | 9 种模板 × 3 语言 + 全功能回归 |

### 4.4 风险管理

## 六、全局高频问题汇总

以下问题在多份文档中反复出现，属于系统性问题：

| # | 系统性问题 | 出现位置 | 根因分析 |
|---|-----------|---------|---------|
| 1 | **越南语技术兼容性** | 需求（排序/搜索）、数据库（collation）、UI（文本长度/字体） | SQLite 默认不支持越南语正确排序，需 ICU 扩展；UI 布局兼容性未验证 |
| 2 | **退货/工单数据回溯** | 需求（成本回调规则缺失）、数据库（回溯字段缺失） | 退货和工单作为后期插入的功能，与主流程的衔接设计不够充分 |

## 九、致项目组的建议

1. **工期是最大风险**：文档质量高但工期偏乐观。建议以 **34 周**作为基线计划，预留 2 周缓冲。尤其关注阶段三B（定制单+生产工单），这是全项目复杂度最高的阶段。

2. **越南语兼容性是技术暗礁**：SQLite ICU 扩展、字体嵌入、UI 文本长度兼容——三者需在 POC 阶段彻底验证，否则后期返工成本极高。

3. **退货/工单是业务短板**：作为后期插入的功能模块，与主流程的衔接（成本回调、数据回溯）设计不够充分，建议优先补全。

4. **尽早引入用户反馈**：建议在 M2.5（采购→入库→库存跑通）后安排越南工厂关键用户试用，将反馈纳入后续迭代优先级调整。

5. **文档体系已达可执行水平**：修复上述高优先级问题后，这套文档可以直接指导开发实施。特别是数据库设计文档的金额精度方案、批次追溯模型和成本双轨制设计，达到了专业 ERP 水准。
