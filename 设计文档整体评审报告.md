## 一、需求规格说明书评审 (`01-requirements.md` v1.4)

### 1.1 核心修复与亮点 (v1.4)

在前次评审后，v1.4 需求文档展现了强大的纠错能力：
1. **补齐了文档版本追踪**：新增了详尽的变更历史。
2. **闭环了财务核算断层**：明确了“退货后移动加权平均成本回调公式”（新平均成本 = (原库存金额-退货金额)/(原库存数量-退货数量)），并对费用分摊尾差采用了标准的“倒挤法”，避免了任何 1 分钱的核算误差。
3. **消除了数据歧义**：金额存储策略已全篇统一指定为 `INTEGER`（最小货币单位），彻底消灭了早先版本中遗留的 `decimal` 用法描述。
4. **统一了枚举体系**：出库类型完全与数据库底层对齐（补充了退货出库和其他出库，达成 4 项）。

## 二、数据库设计评审 (`02-database-design.md` v1.4)

### 2.1 架构级增强 (v1.4)

在此次 v1.4 的重构中，数据库层面消灭了所有 P0 级设计隐患：
1. **全链路可回溯性达成**：`inbound_orders` 与 `outbound_orders` 新增了泛型的 `source_type` 与 `source_id` 字段。至此，销售退货和领料/完工单据逆向追溯有了原生字段支撑。
2. **汇率精度危机解除**：汇率机制由易损失精度的除法（如 0.000039）变更为稳健的反向存储机制（即 1 USD = N 外币）。
3. **生产追溯拼图补齐**：`work_order_materials`（生产领料）正式加入了 `lot_id` 字段，杜绝了生产领料环节批次断带的可能。
4. **数字资产统一性**：表数量与开发计划达成对齐（46 张表），且全部金额模型在设计思路上高度统一。

### 2.2 依然存在的次要改进点 (迭代中处理)

| # | 问题 |
|---|------|
| M1 | `updated_at` 目前仅有建表默认值。因 SQLite 的局限，需靠应用层统一拦截赋值，文档未明显标注防漏策略。 |
| M2 | `system_config` 表作为全局配置，缺少具体的 `backup_time` 或 `setup_completed` 标记参数说明。 |
| M3 | `operation_logs` (操作日志) 若运行 5 年将突破百万行性能拐点，缺少定期归档(Archive)策略描述。 |
| M4 | `stock_checks` (盘点主表) 缺少 `warehouse_id` 后端索引，盘点通常按仓过滤频率极高。 |

## 四、开发计划评审 (`04-development-plan.md` v1.4)

### 4.1 "反盲目乐观" 的拨乱反正 (v1.4)

之前最大的危机在于排期脱离物理现实。v1.4 版的计划完全打消了这个疑虑：
1. **工期基准已校正**：总排期从不切实际的 26-28 周调整为具有高确定性的 **32-36 周**，完全吻合串行全栈开发的合理耗时。
2. **高复杂度节点重估**：全系统最硬核的“阶段三B（销售+定制+库存高级）兼具批次追踪与 BOM 展算”，已被科学地拉长至 6-7 周的攻坚期，防范了项目在中期崩盘的可能。
3. **缓冲规划就位**：明确包含联调、弹性回归（休假/病假等长尾风险缓冲）时间。

### 4.2 剩下的交付把控点

- **测试策略闭环**：计划中强调了 Rust 层核心资产（成本、分摊等逻辑）的 `cargo test` 覆盖，这是单栈开发的最高性价比底线，必须做到严格落实。
- **环境安全网**：Windows WebView2 Runtime 兜底方案和 macOS 公证签名依旧需要在实际构建 Pipeline 中（阶段五）作为最高优先级的攻坚点进行突破。

3. **索引按需动态添加**：开发在撰写关联查询语句 (`stock_checks(warehouse_id)` 等) 时，随手在 migration 补全遗漏。
